# IndigoTest

Тестовое задание: имитаторы бирж + агрегатор котировок.

Разработать имитацию системы сбора, обработки и хранения ценовых данных с нескольких бирж в реальном времени.

## Что внутри

- `IndigoTest.Simulators.First` — первый WebSocket-имитатор.
- `IndigoTest.Simulators.Second` — второй WebSocket-имитатор.
- `IndigoTest.Simulators.Third` — третий WebSocket-имитатор.
- `IndigoTest.Simulators.Common` — общая логика симуляторов: генерация тиков, сбои, статус, WebSocket-цикл.
- `IndigoTest.Aggregators.Web` — точка входа агрегатора и простая страница для проверки.
- `IndigoTest.Aggregators.Operations` — основная логика агрегатора: слушатели, парсеры, транспорт, дедупликация, обработка.
- `IndigoTest.Aggregators.Persistence` — PostgreSQL, контекст, миграция, запись пачками.
- `IndigoTest.Aggregator.Domain` — доменная сущность тика.
- `IndigoTest.Aggregators.Operations.Tests` — тесты на нормализацию, дедупликацию, батчинг, ретраи и переподключение.

### Работа агрегатора
Разделил работу агрегатора на несколько уровней
1. Прослушиватель данных. Блок Sources. На старте SourceListenerHostedService берёт список источников из настроек и запускает прослушивание каждого источника. Дальше SourceListener подключается к конкретному WebSocket через ISourceConnectionClient
При этом сам прослушиватель не должен знать формат каждой биржи. Для этого есть отдельные парсеры
2. Канал данных или транспорт. После парсинга тик не пишется сразу в БД. Он попадает во внутренний транспорт ITickTransport. По-сути хотел заложить в нем поведение Channel
Закладывал то, что в будущем вместо внутреннего транспорта можно было бы поставить брокер, кэш или другой механизм доставки
3. Процессор данных. ITickProcessor
Основная часть обработки нормализованных данных. Он читает данные из транспорта, прогоняет через дедупликатор и собирает батчи для записи
Дедупликатор вынесен в ITickDeduplicator. Сейчас он работает через ключ, собранный из важных полей котировки. Если такой тик уже был в пределах окна дедупликации, он считается дублем и дальше не пишется
4. Сохранение данных. После обработки данных нужно сохранить данные. Это отдельый процесс.
Сохранять данные мы можем в разные приемники, поэтому сделал ITickWriter с реализацией PostgreSqlTickWriter
В рамках него не использовал трэкинг данных от EF, а использован бинарную вставку. Описал ниже.
Система по-сути сейчас не идеальна, т.к. мы работаем только с одной реализацией без каких-либо policy выбора нужного Writer'а
5. Мониторинг данных. SourceListener отвечает за жизненный цикл подключения. Если WebSocket оборвался, источник закрыл соединение или произошла ошибка, слушатель не умирает навсегда, а ждёт задержку и пытается подключиться заново
SourceReconnectPolicy отвечает за задержку между попытками
SourceLivenessMonitor отвечает за ситуацию, когда сокет вроде живой, но данные перестали приходить
Для сбора и отображения метрик есть IAggregatorStatus. Он по-сути отвечает за хранение счетчиков сколько тиков принято, обработано, отброшено из-за дублей, записано, сколько в очереди и отброшено

## Форматы бирж

Форматы по ТЗ разные

Первый источник

```json
{
  "symbol": "BTCUSDT",
  "price": 67000.15,
  "volume": 1.25,
  "timestamp": "2026-07-05T10:00:00+00:00"
}
```

Второй источник

```json
{
  "s": "ETHUSDT",
  "p": "3450.42",
  "v": "2.75",
  "ts": 1783245600000,
  "kind": "ticker"
}
```

Третий источник

```json
{
  "instrument": "SOLUSDT",
  "trade": {
    "last": 152.30,
    "qty": 4.125
  },
  "meta": {
    "time": "05.07.2026 10:00:00.123",
    "source": "third"
  }
}
```

Внутри агрегатора все приводится к одному виду:

```text
source, symbol, price, volume, timestamp, receivedAt
```

## Дедупликация

Дубликатом считается тик с одинаковыми:

- `Source`
- `Symbol`
- `Price`
- `Volume`
- `Timestamp`

Окно дедупликации: `1 минута`
 симуляторы при дублировании отправляют тот же самый тик второй раз. Для такого тестового этого ключа достаточно. `ReceivedAt` в ключ не входит, потому что повтор может прийти позже

## Переподключение

Агрегатор держит отдельное WebSocket-подключение к каждому источнику

Если один источник упал или закрыл сокет:

- остальные источники продолжают работать
- проблемный источник переподключается отдельно
- задержка переподключения растет по экспоненте
- максимум задержки сейчас `30 секунд`

Еще есть idle-timeout. Если сокет формально живой, но данные перестали приходить, агрегатор не ждет бесконечно. Он считает соединение зависшим и переподключается

## Запись в БД

Реализован батчинг данных. Размер батча - управляется через параметры приложения
Посчитал, что для потоковых данных применение трекинга будет сильно излишним
Корректнее реализация - через бинарную вставку, которую поддерживают многие СУБД. Применил Binary COPY от PostgreSQL

сценарий, где БД временно не отвечает:
- текущий батч не выбрасывается
- ретрай с задержкой
- транспортный канал ограничен по размеру
- когда канал заполняется, чтение из источников начинает ждать
- не добавленные данные - дописываются в БД в порядке очереди.

## Graceful shutdown

При остановке агрегатора источники завершаются, транспорт закрывается, а обработчик дописывает то, что уже успело попасть в очередь
